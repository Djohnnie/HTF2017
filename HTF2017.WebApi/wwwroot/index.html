Anonymous Androids

We are 2030 and the world has reverted into chaos. 766 million of the poorest people are roaming the planet with no home and in search of a better life. Because it is hard to infiltrate or hack their complicated communication and voting network without them knowing, another way of infiltration was devised. A large number of highly advanced androids, indistinguishable from actual humans, are being released onto the general public as we speak. These androids will anonymously collect data that we should process to anticipate any unexpected events or attacks against the established order.

Get started

Browse to the HTF2017 .NET challenge dashboard to create your team and get started!

Technically, this challenge is about communicating with an HTTP webservice using .NET code and writing your own HTTP webservice using ASP.NET WebAPI. You need to write .NET client code to call an existing HTTP webservice (written using ASP.NET WebAPI) to issue commands. You need to write .NET server code using ASP.NET WebAPI to receive feedback from the server. Additionaly you need to work on a dashboard in a UI technology of your choice.

Androids

Your job is to deploy androids and create a dashboard to follow-up on their actions and findings. Each team has access to 1.000 of our newest and most advanced androids. Their workings are simple, but they can give you a lot of information about our threats.

There are three kinds of androids:

  • Level 1
    These high-level androids are completely autonomous and decide by themselves if it is safe to send you their sensory data. They have a very low chance of being detected of infiltrating and are therefore very reliable. Their sensor updates are automated but not very frequent based on the increased security risk. Sending manual requests is not supported! The sensory data is low in frequency and usefullness, but the android is likely to last a long time in the field gathering some useful data.
  • Level 2
    These medium-level androids are partially autonomous and decide by themselves if it is safe to send you their sensory data. You have the option to request a sensor update with the risk of the android getting detected. Their sensor updates are automated just like the high-level androids. The sensory data is low in frequency and medium in usefullness. The android is likely to get detected if you request some sensor data.
  • Level 3
    These low-level androids are partially autonomous but never decide to send you their sensory data. You need to request updates with the risk of the android getting detected. The sensory data can be high in usefullness. The android is very likely to get detected if you request some sensor data with high accuracy.

Sensors

Each android has a limited number of sensors that can provide you with useful data.

  • Location
    Returns the location based on longitude and lattitude. If the sensor is off, no location is measured. Depending on the sensor accuracy the location accuracy has a 1km fault radius for high accuracy, a 100km fault radius for medium accuracy and a 1000km fault radius for low accuracy.
  • Crowd
    Returns the size of a crowd based on number of people. If the sensor is off, no crowding is measured. Depending on the sensor accuracy the crowding accuracy has a deviation of 100 people for high accuracy, 1K people for medium accuracy and 10K people for low accuracy.
  • Mood
    Returns the mood of a crowd based on an 8-bit number where a low number describes a relaxed mood and a high number describes a tense mood. If the sensor is off, no mood is measured. Depending on the sensor accuracy the mood accuracy has a deviation of 16 for high accuracy, 64 for medium accuracy and 128 for low accuracy.
  • Relationship
    Returns the relations between people of a crowd as an 8-bit number where a low number indicates no relations between people and a high number indicates strong relations between people. If the sensor is off, no relationship is measured. Depending on the sensor accuracy the relationship accuracy has a deviation of 16 for high accuracy, 64 for medium accuracy and 128 for low accuracy.

ngrok

ngrok is an application for Windows, Mac OS X, and Linux that creates a tunnel but also allows you to inspect all traffic that goes through the tunnel and replay that traffic for testing.

You can download ngrok, unzip the package and run it from command-line or PowerShell.

To make ngrok compatible with IISExpress (Visual Studio Debugging of ASP.NET web applications) you should add your ngrok hostname to the website bindings of the IISExpress configuration. For this you need to find the IISExpress icon in your Windows notification area, right click it and choose "Show all Applications". In this new window, you click your application and choose to edit the configuration in the bottom area of the window. In the configuration file you search for the <bindings> element and add the ngrok hostname to the bindings list to make ngrok work.

Your plan of attack!

Our suggested plan of attack is:

  1. Register a team using the dashboard website.
  2. Create an ASP.NET WebAPI project that is able to process GET requests without arguments and return a HTTP OK.
  3. Deploy, or locally run your ASP.NET WebAPI
  4. Setup ngrok to make your locally deployed webservice available from the internet.
  5. Update your team feedback URL with the public ngrok URL using the dashboard website.
  6. Test the feedback URL using the dashboard website.
  7. Extend your ASP.NET WebAPI project to also have a POST ability on the same URL and make it process the SensoryData feedback data and return your team identifier to acknowledge you have received it successfully.
  8. Deploy your first level-2 or level-3 bot and request a Sensory Data update.
  9. Watch your ASP.NET WebAPI and team score to identify if the Sensory Data is received.
  10. do-while until you are satisfied with your server code.
  11. Create a dashboard to show the data that has been received by your bots. The prettier the better!
  12. Create a control center UI to help you deploy androids and request updates. The more user friendly, the better!
